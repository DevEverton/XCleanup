# XCleanup — Design Spec

**Date:** 2026-07-02
**Status:** Approved. Amended later the same day: sandbox dropped in favor of
open-source direct distribution (see "Distribution & permissions").

## Problem

Xcode and related dev tooling accumulate build artifacts fast. On this machine, they reached ~340 GB of logical size before cleanup: 202 GB of per-package SwiftPM `.build` folders inside a project repo, 49 GB of root `.build`, 40 GB of DerivedData (in-repo + global), 46 GB of XcodeBuildMCP workspaces, 46 GB of simulators, 11 GB of iOS DeviceSupport. There is no at-a-glance visibility and cleanup is manual terminal work.

## Product shape

A **macOS menu bar app with a companion window**, built with SwiftUI.

- `MenuBarExtra` scene, `.window` style; `LSUIElement = YES` (no Dock icon).
- Clicking the icon opens a panel listing each artifact category with its current size and a Clean action.
- Sizes refresh when the panel opens: cached values render instantly, live scan results replace them as they complete. A manual refresh button forces a rescan.
- Deployment target: macOS 26.2 (as generated by the project template).

### Panel design (amended 2026-07-02, research-driven)

Modeled on Stats/DevCleaner patterns (see `.lazyweb/design-research/menubar-panel-and-window-2026-07-02/`):
hero header with an animated "total cleanable" figure and a stacked proportion
bar tinted per category; rows use tinted icon chips, hover highlights and
hover-revealed Clean buttons, snappy expand animations, monospaced-digit sizes,
and a green "Clean" state when a category has nothing to reclaim. Footer:
rescan, last-scan time, open-window, settings, quit.

### Companion window (added 2026-07-02)

`Window` scene (id `main`) opened from the panel footer (`openWindow` +
`NSApp.activate`). `NavigationSplitView`: sidebar lists Overview + the five
categories with tints and live size badges; Overview shows the hero total,
stacked bar, disk-free context and per-category cards; each category detail is
a `Table` (Name | Location | Size) with multi-selection, context menu,
toolbar actions (Rescan, Delete Selected, Clean All, Delete Unavailable for
simulators) and a Finder-style status bar (item count · total · selection).
All destructive actions remain behind confirmation dialogs.

## Cleanup targets

A `CleanupTarget` protocol — `name`, `scan() async -> ScanResult`, `clean(selection) async throws` — with five implementations:

| Target | Shows | Clean action |
|---|---|---|
| DerivedData | `~/Library/Developer/Xcode/DerivedData` total + per-project rows | Delete all or per-project |
| XcodeBuildMCP | `~/Library/Developer/XcodeBuildMCP/workspaces` total | Delete all |
| iOS DeviceSupport | `~/Library/Developer/Xcode/iOS DeviceSupport` per-OS-version rows | Delete selected versions |
| Simulators | Every device under `~/Library/Developer/CoreSimulator/Devices`: name, runtime, size, availability (parsed from each device folder's `device.plist`) | Delete device (remove folder), Erase (wipe `data/` contents), one-click "Delete unavailable" |
| SwiftPM .build | Scans user-added project roots (e.g. `~/Desktop/dev`) for `.build` dirs, grouped by repo/package | Delete all or per-package |

### Safety rules (all targets)

1. Deletions only occur inside user-granted roots.
2. Only paths matching the target's known pattern are eligible; a `.build` directory is eligible only if a `Package.swift` or `Package.resolved` sits beside it.
3. Symlinks are never followed during scan or delete.
4. Every destructive action is behind a confirmation dialog.

## Distribution & permissions

**Amended 2026-07-02:** App Sandbox is **off**. The project is open source
(MIT) and distributed directly (GitHub releases / build from source), not via
the Mac App Store. Hardened Runtime stays on so Developer ID notarization
remains possible.

- No grant flow and no onboarding: `~/Library/Developer` is read directly.
- Project scan roots live in `ScanLocations` with two modes (amended
  2026-07-02): **Automatic** (default — scans the home folder; the scanner
  skips system folders like Library, Applications, Movies, Music, Pictures,
  Public, .Trash) and **Specific folders only** (user-managed path list, for
  external drives or faster scans). Settings shows a radio picker; the folder
  list is only visible in custom mode.
- macOS TCC still prompts once for protected folders (Desktop, Documents,
  Downloads) when first scanned; denial degrades gracefully to empty results.

Original design kept the sandbox with security-scoped bookmarks for App Store
eligibility; that flow was implemented and then removed in the same day's
pivot. Retained trade-offs of file-level simulator management (no `simctl`):

- CoreSimulator notices deleted devices on its next service restart.
- "Erase" is a best-effort wipe of the device's `data/` contents, not `simctl erase`.

## Architecture

- `AppState` (`@Observable`): owns the target list, per-target `ScanResult` (total size, item breakdown, timestamp), scanning/cleaning flags. Scans run concurrently in a `TaskGroup`, cancellable, off the main actor.
- `SizeScanner`: `FileManager` enumerator summing `totalFileAllocatedSizeKey` (physical size — immune to APFS-clone over-reporting).
- `BookmarkStore`: persistence and resolution of security-scoped bookmarks.
- Views: `MenuPanelView` (category list → expandable rows → per-item rows with Clean buttons), `OnboardingView` (grants), `SettingsView` (project folder list, launch-at-login via `SMAppService`, background refresh interval).
- Background refresh: a timer rescans all targets every 6 hours by default (configurable in Settings, can be disabled) so the panel opens with recent cached sizes instead of stale ones. No size text in the menu bar label — just the icon.

### Data flow

grant → bookmark stored → scans (TaskGroup) → results published to UI → clean action → targeted rescan of that category only.

## Error handling

- Partial failures during clean (file in use, permission denied mid-tree) are collected per item and shown inline; the clean continues past them, and the post-clean rescan shows ground truth.
- Missing directories display as "—", not errors.
- Bookmark resolution failure re-prompts via the grant flow.

## Testing

XCTest unit tests against fixture trees built in `NSTemporaryDirectory()`:

- `device.plist` parsing (name, runtime, UDID, availability).
- `.build` discovery rules, including the Package.swift-sibling guard and symlink refusal.
- Size scanning correctness.
- Deletion selection logic (what would be deleted for a given selection).

UI is a thin layer over `AppState` and is verified manually.

## Out of scope (YAGNI)

- Auto-clean on threshold / scheduled deletion.
- `simctl` integration or a privileged helper.
- Cleaning caches outside the defined targets (Archives, CocoaPods, npm, etc.) — can be added later as new `CleanupTarget` conformances.
